# Pilot — observatiechecklist (intern team)

**Doel:** tijdens een live testsessie vastleggen waar het product **wrijving** geeft, zonder de tester te onderbreken.  
**Gebruik:** één checklist per sessie; vul kort in (steekwoorden).

---

## Voor de sessie

- [ ] Testaccount en rol (gemeente / zorgaanbieder) zijn bekend bij de tester.
- [ ] Tester weet dat **observatie** geen beoordeling van hem/haar is, maar van het systeem.
- [ ] Schermopname alleen met **toestemming** en volgens interne privacyregels.

---

## Tijdens de sessie — signalen

| # | Waar let je op? | Ruimte voor notities (steekwoorden) |
|---|-----------------|-------------------------------------|
| 1 | **Aarzeling** — stopt de tester >5 seconden zonder duidelijke reden (lezen vs. zoeken)? | |
| 2 | **Verkeerde verwachting** — doet een knop iets anders dan de tester dacht? | |
| 3 | **Dubbele of onduidelijke hoofdactie** — twee knoppen die allebei “het belangrijkst” lijken? | |
| 4 | **Label** — woord dat de tester hardop corrigeert of vraagtekens bij zet? | |
| 5 | **Ontbrekende context** — “waarom sta ik hier?” / “voor wie is dit?” | |
| 6 | **Fase / status** — begrijpt de tester **waar** de casus in de keten staat? | |
| 7 | **Eigenaar / verantwoordelijke** — is duidelijk **wie** nu moet handelen? | |
| 8 | **Volgende stap (NBA)** — is er **één** duidelijke volgende actie, of moet de tester zoeken? | |
| 9 | **Leeg / fout / laden** — voelt een lege staat uitlegend of “kapot”? | |
| 10 | **Zorgaanbieder-zicht** — ziet de tester iets dat **niet** bij zijn/haar rol past? | |

---

## Direct na de sessie — 5 minuten met tester

- Wat was het **duidelijkste** moment?  
- Wat was het **minst duidelijke** moment?  
- Welk **één** zou ze als eerste willen verbeteren?
